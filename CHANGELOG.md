## [Unreleased]

### Fixed

- Fixed concurrent sync passes interleaving over one addressbook: the in-app sync, the background worker and the OS-scheduled sync adapter could each run an engine pass over the same book at once, and a pass spans many store transactions, so interleaved passes could double-push staged creates or clobber the sync checkpoint; a process-wide per-book lock now serializes them, other books still syncing in parallel.
- Fixed a schema rebuild destroying the contacts nothing can restore: upgrades dropped the card tables wholesale, taking the On this device book's cards (their only copy) and every unpushed edit and staged deletion with them, and installing an older build over a newer store crashed outright. Rebuilds now carry those rows over (only the columns both schemas share, so any version pair works in both directions), clear the surviving sync checkpoints so the next round enumerates in full instead of silently never re-downloading the dropped cards, and downgrades rebuild the same way instead of throwing.
- Fixed two concurrent OAuth token refreshes able to log an account out for good: the credential store's read-modify-write had no lock, so a foreground sync and a background worker refreshing at once interleaved last-writer-wins and could persist a stale rotated refresh token. The store now serializes process-wide, and the activity's in-memory account cache only mutates on the main thread the drawer reads it from.
- Fixed a sync finishing after the screen rotated away crashing the app: the completion callbacks (loader dismissal, report toasts, the error dialog) ran against the destroyed activity's dead window (BadTokenException); they now only run while the activity is alive. The OAuth loopback listener also closes with the activity, where it used to keep its port blocked for up to five minutes (interrupting its thread cannot unblock accept, only closing the socket can).

- Fixed every server-axis write wiping the card's phone axis: setMembership wrote the membership row with INSERT OR REPLACE, and SQLite REPLACE deletes and reinserts the row, so phone_revision and phone_base nulled out each time a server pass merely touched the card; the next phone pass then saw every touched card as never-projected, whose raw contact already existed, and re-created or conflicted the lot (confirmed on-device as the driver of the mass local-change waves). The membership state now updates in place, leaving the axis alone.
- Fixed a permanent refresh loop against servers whose fetch ETags differ from their listing ETags (posteo's SabreDAV for a subset of cards; Google People's people.get flavour likewise): the engine's refresh recorded the listing ETag, hydration overwrote it with the fetch ETag, and the next listing mismatched again, re-downloading the same cards on every sync forever (and re-triggering the axis wipe above). Fetched items now record the ETag the pass's own enumerate listed, falling back to the fetch flavour only when the handle was never listed.
- Fixed the phone spoke fabricating a content change for every raw contact it read back: the fetch boundary re-encoded the last converged vCard through applyCard even when the field-space diff took nothing from the phone, and since applyCard rewrites every managed property in canonical form (order, parameter case, folding), the re-encoded bytes hashed differently, the engine adopted them as a phone edit, marked the row dirty against the server and pushed it upstream on the next pass; one stale revision across the set (an app update changing the revision scheme was enough) thus rewrote the whole book locally and then remotely. A fetch that takes no field now returns the converged vCard byte for byte, so it registers as content-quiet on both axes.
- Fixed pulled phone edits never converging provider-side: nothing on the pull path cleared the raw contact's DIRTY flag or stamped SYNC1 (only pushes did), so a contacts-app edit, once ingested, left its row in the dirty:VERSION sentinel regime forever and every provider-internal VERSION bump (aggregation, name recomposition) re-fetched it; raw contacts projected before the SYNC1 scheme existed sat in the same regime. Every read now stamps the row converged, guarded on the VERSION observed before the data rows were read so an edit racing the ingestion loses the stamp and is picked up next round.
- Fixed the phone read-back listing data rows in provider order: the field-space diff compares multi-valued fields as lists, so an unordered listing could read as a permutation edit; the query now pins insertion order.

- Fixed removing an account leaving its phone (Android) accounts behind: the delete now removes each of the account's book-backed phone accounts explicitly (ContactsProvider purging their raw contacts with them) and then reconciles against the remaining phone-synced books, the same set the sync path maintains, where it used to reconcile against every subscribed book and count on the absence-purge alone.
- Fixed the minified builds (every release build, so every APK users actually install) failing all network operations with a JNI method-not-found error, the discovery screen landing on the not-found row plus the Google/Microsoft fallback chooser. The R8 keep rules described an older Transport API (connect, read and write without the URL parameter), so they matched nothing and R8 stripped the real read(String) and write(String, byte[]), which only native code calls; the rules now pin the actual signatures and also cover OfflineDriver.serve, the offline engine's upcall, which had no rule at all. Debug builds do not shrink, which is why development installs never showed it.

### Changed

- The contacts list no longer blocks the main thread to rebuild: the full-table scan and the bridge grouping behind every reload moved into a ContactPool running on the background executor, the render landing back on the UI thread, and searching now filters the already-grouped rows instead of re-grouping per keystroke. The import/export text handling became a pure Vcf codec (split and join) pinned by its own tests, and the duplicate remover and the next-birthday peek moved into their own classes (DuplicateReview, Birthdays), talking back to the screen only for the merge flow and the editor. The editor's dozen loose state fields became one EditSession value object, built fresh when a contact (or merged group, or conflict) opens and replaced blank when the editor leaves. The screen navigation collapsed onto one table: each screen's chrome, FAB action and back behaviors live in a single Screen entry, replacing the four parallel switches that each had to know every screen. The bridge's client API split along the transport line: the pure vCard computations and the io-offline plan verbs moved onto a static Cards class, so the card store, the edit form, the contacts pool and the phone spoke no longer construct a network-capable client just to index, project or plan. The card store split the same way: the io-offline storage seam (placement loading, engine writes, conflict bookkeeping) moved into an OfflineStore over the same SQLite helper, leaving CardStore the app-facing store (addressbooks, cards, memberships, the merged view's link layer) plus the shared row vocabulary. The contacts screen itself (the merged list, its recycling adapter, the in-bar search and the multi-select mode) moved into a ContactsList class, leaving the activity the shared bar and the flows that leave the screen; the main activity dropped from roughly 5,850 lines to under 3,200 across the whole refactor.
- The connection wizard and its OAuth grants moved out of the main activity into their own classes: OnboardingFlow carries the email step with its parallel discovery, the config proposals, the credential prompts and the addressbook selection whose commit persists the account and runs the first sync; OauthFlow carries the shipped Google and Microsoft grants, custom clients over the loopback or app-scheme redirect, the zero-registration issuer path and the persisted in-flight grant, landing redeemed tokens back in the wizard through a connector callback. The activity keeps the step navigation (the flipper, the bar, the shared FAB) and the intent routing; the shared theme and density helpers the screens duplicated live once in a Ui class now. Pure repackaging: no behavior changed.
- The sync run now has one owner: a SyncRunner behind every entry point (the in-app sync, the background worker) carries the account lookup, the addressbook self-heal and the refresh-once-and-retry dance that used to exist twice, aligned only by a comment; the activity keeps the loader dialog, the toasts and its in-memory account cache (fed by an observer), the worker keeps the scheduling and the notification. The riskiest Java logic also gained its first engine-level tests: Robolectric runs the real SQLite store combined with the real Rust bridge built for the host, pinning the membership staging state machine, the schema-rebuild carry-over, a pulled card landing through the same write path the engine drives (the phone axis surviving a server write included), and the driver's serve wire contract.
- Errors now cross the Rust-Java bridge structured: every error reply carries the HTTP status of the failed round as its own field next to the message, CardamumException exposes it (surviving the round trip through the offline engine's driver seam), and every decision that used to substring-match the message prose (the CardDAV If-Match quirk retry on 412, delete convergence on 404, the token refresh on 401, Google's etag-representation retry on 400) branches on the status instead, so a reworded upstream error message can no longer silently disable them.

- The conflict-resolution form became the plain editor itself: opening a diverged contact shows every field and keeps the add-field controls (the bar button and the section-header adds), instead of a reduced page limited to the disagreeing rows. Two signals remain: each diverging row wears the error-tinted diverged glyph until opened and confirmed (rows added during the session never wear it), and while any glyph remains the save disc turns the error tone and wears the same warning glyph, full strength rather than a dimmed check, refusing to validate. Review keys moved from page positions to field identities, so adding or removing entries around the conflicts no longer risks misplacing the warnings; and a reviewed row follows the plain rules again, so clearing its value drops its line exactly like in a normal edit (it used to stay as an empty row for the rest of the session).
- The edit form's list rows now render like its single rows: the type (Mobile, Home) or the item kind (Nickname, Website, Note) on the title line and the value beneath, so phones, emails and the other lists share the single fields' label-over-value scale instead of showing their values at label size.
- The app settled on one type scale: primary text at 16sp (contact names, form property titles, chooser and option rows, dropdown and spinner values, onboarding paragraphs), secondary text at 13sp (list info lines, form values, captions) and the edit form's accent section titles at 13sp; the list's 16sp names, the form's 18sp titles and the assorted 13sp-16sp labels all folded onto it.
- Replaced the sync spinners with a modal progress dialog: any sync action (drawer, pull-down, Sync local, the post-onboarding first sync) opens the same non-dismissible dialog (no back, no outside tap), a spinner beside the addressbook being synchronized and the current engine step under it (checking the server, downloading or sending n contacts, reconciling with the phone, resolving n conflicts), the screen staying on until the sync ends, so a long sync is explicit about what it does instead of an ambiguous spinner. The dialog carries the only spinner anywhere: the FAB keeps its plain icon, the pull-down gesture only triggers the sync (its own spinner retracting right away) and the drawer's sync row slides the drawer shut as it launches (the dialog runs over the contacts list, the outcome toasts land on the refreshed list) and just goes inert while a sync runs.
- The phone pass now takes a quiet path when there is nothing to do: one indexed provider count (no dirty, new or deleted raw contact), one store EXISTS (nothing staged hub-side) and a member-count match replace the whole engine pass, so a no-change sync costs the server round trip plus a few queries. When the pass does run, enumerate reads each contact's sync state from its own cursor instead of one provider query per contact.
- CardDAV pushes now run concurrently (four in flight) instead of one blocking round trip per card: the protocol has no batch verb (RFC 6352 only batches reads via addressbook-multiget), so parallel guarded PUTs and DELETEs over per-call connections are the lever, and a 150-card round drops from minutes to well under one.
- Google People creates and deletes now group into the batch verbs (people.batchCreateContacts at 200 a call, people.batchDeleteContacts at 500) instead of one call per card: an import of 150 contacts is one request instead of 150, which also stops large rounds from tripping the per-minute write quota; guarded updates and membership patches stay per-card.
- JMAP rounds now ride ContactCard/set calls of 50 changes (creates, content updates, membership patches and destroys in one request) and Graph rounds ride $batch calls of 20 inner requests, both mapping the per-object response back to per-change results: a rejected card reports rejected for the next sync to reconcile instead of aborting its whole round, and a destroy the server no longer finds reads as already converged.
- The auth flow's first step titles itself Welcome only on a fresh install; with an account already stored it reads Add account instead.
- The auth flow now appears and leaves with a fade and a slight zoom, settling down from slightly larger on entry and rising back on exit (plain framework view animations, shared with the account settings screen) instead of the drawer-like slide, and opening the app with no account shows it right away instead of after a half-second beat.
- The per-addressbook settings dialog gave way to a full-screen account settings screen, opened from the drawer's account rows with the auth flow's fade-and-zoom: its own bar carries the account email, a closing cross landing back on the open drawer (the screen raises itself above the drawer, which stays open underneath, so closing it, saving or deleting always reveals the account list again) and a trash deleting the account after confirmation (replacing the old header long-press; the confirm is a titled dialog asking "Are you sure you want to remove this account?", without repeating the email). The form is uniform checkbox rows on the edit form's scale (48dp height, 16sp labels, 16dp start and 12dp end padding, so the trailing boxes centre on the bar buttons' 28dp icon centreline): Enable this account and the background-sync cadence dropdown, both fanning out onto every addressbook of the account (most providers expose a single one), then, past a 1dp separator inset to the rows' paddings, Enable advanced settings per addressbook unfolding one section per book (its name as a plain header, then Enable this addressbook, Sync remotely, Sync locally and its own cadence, indented and dimming while the book is off). Nothing persists as fields change; the screen's own check FAB commits the whole staged form at once. The store gained the per-book remote switch behind Sync remotely: a book can keep its contacts in the app, and even mirror them to the phone, while its server exchanges pause (the sync then runs only its phone pass).
- The edit form's section separators slimmed to a 1dp line (still the app bar tone) sitting flush: the last item's own row padding is the gap now instead of stacking onto a line margin, and a revealed-but-empty section pads its header instead, so every separator gets the same clearance.
- The in-app sync report now shows one toast per axis, Local first then Remote (the pending-conflicts line rides the remote toast) instead of both lines on one toast; the background notification keeps both axes on one card.
- The drawer became a plain list of accounts (user feedback: the account-and-addressbook tree was too much, and most providers expose one addressbook anyway): one row per connected account, a leading account glyph, the email and a trailing chevron, tapping anywhere on the row opening the account's settings screen; the shared item text size went from 13sp to 16sp, so the account rows and the footer actions read the same. The title bar swapped its account icon for a leading closing cross, in the shared nav-icon-first bar shape; the account folds, the per-book rows with their sync-reach glyphs and the long-press delete all left with the tree, and the footer keeps its 1dp surface-tone delimiter above Sync, Add account and About.
- The contacts overflow menu went text-only (the framework popup renders forced icons flush against their labels on some Android releases, stock Android 13 for one) and slimmed to Import and Export: Synchronize left the menu (the drawer, the pull-down and the modal loader carry sync), Find duplicates moved onto its own app-bar icon next to the menu, and the menu button wears the import-export up-down arrows instead of the three dots to match what is left. The birthday and duplicates icons stay put through the in-bar search, the field shrinking to end at them like it already did at the menu, and the search pill took the view-background tone, so it reads as a field against the surface-toned bar.
- Raised the minimum supported Android version from 7.0 (API 24) to 8.0 (API 26). Android 8.0 is the first release whose system trust store ships the ISRG Root X1 certificate, so Let's Encrypt-certified servers (most self-hosted CardDAV instances) validate against the platform trust the transport relies on; 7.x devices have failed those handshakes since the cross-signed chain expired. The pre-8.0 fallbacks (the channel-less notification path, the colorError substitute) went with the floor.
- Moved the app chrome (the app bar, the drawer header and bottom band, the auth and settings panel headers, the Google build's support action bar, and the status bar) off the device theme's colorPrimary onto the app's neutral surface colour. DeviceDefault's colorPrimary resolves to an arbitrary vendor tone (a saturated brand blue on many devices, a wallpaper-derived pastel on others) that the textColorPrimary foreground pairs badly with; the surface tone reads with it on every vendor and version and follows the light/dark setting, the modern neutral-top-bar look. The two spinners that rode colorPrimary now tint with the accent like every other tinted control, and the editor's section separators follow the chrome onto the surface tone.
- The sync report (the toast and the background notification) now reads per axis and counts distinct affected cards: a Local line (n in, n out, n changed, against the phone's Contacts app) when any synced book mirrors there, a Remote line for the server, and pending conflicts appended as their own line rather than replacing the report. In means a card appeared, out a card was deleted, changed a card whose content changed, either direction, and a card counts once however many passes touched it: the driver tallies accepted pushes and effective store writes into per-card sets, so the multi-pass echo that used to triple-count a single edit (each phone projection bumps the raw contact VERSION, which the next pass re-reads and re-merges) reports as the one changed card it is. The background notification is titled by the account email, the addressbook name joining it when expanded.
- Replaced the Google Play subscription paywall with the earnest support prompt of docs/monetization.md: the Play build is now free, complete and ungated like every other channel, and no launch-time entitlement gate or cached subscription check remains. Instead, opening the app shows a dismissable full-screen ask at most once per rolling 3 days (only on genuine user-initiated opens: never over an OAuth redirect, a headless sync hook, or a rotation; a fresh install waits 3 days before the first ask), explaining that continued development depends on the people who use the app. The panel is framed as the app's price list rather than a donation banner: a statement of fact ("Cardamum is paid by the people who use it"), the pay-once-and-this-screen-never-shows-again deal line, and the one-time pay-what-you-want tiers as radio options (Supporter 5, Backer 10, Patron 25, Sponsor 50, Benefactor 100 EUR), each with its description visible below and the cheapest checked by default (live Play names, prices and descriptions when reachable, static labels otherwise); a fixed bottom bar carries the LATER dismiss and the accent pay button, whose label follows the selected tier into the purchase sheet; paying any tier silences the prompt permanently (also recognized when paid on another device or before a reinstall, in which case the panel closes itself unseen), and dismissing (a small Later button, or back) just closes it until the next window. The FOSS builds (GitHub CI, F-Droid) keep carrying no prompt and no Google code.

### Added

- Added a next-birthday peek, a cake icon in the app bar between search and the duplicate remover: one tap tells, in a single sentence, who celebrates the next birthday and when ("Alice Dupont and Bob Martin celebrate their birthdays on 12 April, in 3 days."). Birthdays come from the merged cards' projected BDAY (full dates only), computed off the UI thread; with none recorded the dialog says so.
- Added the standard-or-advanced choice to the connection flow: submitting the first field pops a dialog (while the discovery already runs behind it) offering the standard setup or the advanced one. Standard runs the first proposal directly, skipping the config screen; a shipped OAuth client (Google, Microsoft) also skips its one-tap confirm dialog and goes straight to the browser, while a password or API token is still asked. Once connected it commits without further steps: every addressbook subscribed with phone mirroring and a 15-minute background sync, straight into the first sync and the contacts list. Advanced keeps the step-by-step flow (configuration choice, addressbook selection, cadence). A discovery that found nothing shows the fallback chooser even in standard mode, since auto-signing into a guessed provider would be wrong.
- Added scheduled background synchronization, the DAVx5 way (WorkManager, not a daemon: Android offers none). Every addressbook carries a "Synchronize in background" cadence (never by default, then 15 minutes up to daily), set once for all selected books at the end of the connection flow and tuned later from the account's settings screen (the whole account at once, or per book behind Advanced); each enabled book gets one unique periodic worker running the same three-spoke hub pass as the in-app Sync (phone, server, phone), network-constrained, self-healing on app launch and self-cancelling when the book or its schedule goes away, with an expired OAuth access token refreshed and re-persisted like the in-app sync. A pass that did something posts a system notification (the account email as the title, the addressbook name joining it when expanded, and one line per axis counting affected cards: Local in/out/changed when the book mirrors into the phone, Remote likewise; its own channel, tap opens the app, replaced in place by the next report); a pass with nothing to report posts nothing. Contacts in a both-sides-edited conflict sit each pass out, exactly as in the in-app sync (the engine parks them untouched, per item, no merge and no push, while everything else keeps syncing; a remote delete of a conflicted contact resurrects the local edit rather than dropping it), and pending conflicts ride the notification as a warning subtitle, re-raised every pass until they are resolved in the app. Enabling a cadence also turns on the Android account's content-triggered sync, so a contacts-app edit uploads into the hub as it happens and rides upstream on the next periodic pass; the onboarding confirm asks the needed permissions up front in one grouped request (contacts for the default phone mirroring, sparing the ask at the first sync, plus notifications on Android 13+ when a cadence is on), and enabling a cadence later from the account settings screen still asks for notifications.
- Added the two-way phone spoke: the phone's contacts are now a second io-offline collection per addressbook (docs/phone-sync-plan.md), reconciled by the same engine as the server, so edits, creations and deletions made in any contacts app on Cardamum's raw contacts converge into the store and push upstream, and vice versa. Each sync runs phone, server, then phone again, so a phone edit reaches the server in one pass; "Sync local" runs the phone pass alone, in-process behind the same spinner (no sync-scheduler detour), while the registered sync adapter serves the syncs the OS schedules on its own: the system's per-account "sync now" and the upload syncs Android requests right after a contacts-app edit. The read-back is a field-space patch onto the last converged vCard (only the fields the phone actually edited overwrite the model), so properties the phone cannot represent survive every round-trip; divergences three-way merge like server conflicts, an update beating a delete. The one-way projector is gone, and the store schema bumped to v15 (the phone axis lives on the membership rows, per book).
- Added zero-registration OAuth onboarding: a discovered config that carries only an authorization-server issuer (from a provider's PACC oauth-public block) now offers OAuth, and picking it fetches the issuer's RFC 8414 metadata, registers a public client on the fly via RFC 7591 (loopback redirect, no secret) when the server allows it, and runs the usual PKCE code grant against the discovered endpoints, persisting the account with the issued client id so refresh keeps working; a server without a registration endpoint (Google, Microsoft) falls back to the custom-client prompt with the endpoints prefilled. Fastmail is the first provider this onboards with no pre-registered client at all.
- The advanced editor's property form proposes the RFC 6350 registries as dropdowns: the property name and each parameter name are completing fields over the known names (free text still allowed, the dropdown opening on the very first tap), and the parameter values complete per comma-separated token against the named parameter's own registry (the TYPE values, the VALUE kinds, CALSCALE). The value area also shapes itself from the property name: structured values (N, ADR, GENDER) show one labeled field per component, decoded and recomposed with proper escaping through the bridge (the labels mirroring the vcard-rs value types), everything else keeping the raw field; and the "Add property" row took the edit form's "Add <field>" style, plus icon included.
- The advanced editor became two-level and structured: each property row now opens a real form (the property name, one editable row per parameter with its name and comma-separated values plus an "Add parameter" row, and the raw value), recomposed and validated by the bridge (separators are stripped from parameter values: quoting would not survive the parser's round trip). The source block at the bottom is now tappable and opens the second level, a full-screen free-hand vCard source editor whose check FAB applies the text after a reparse validation.
- Every edit-form section header now carries its icon in front of the accent label (person, label, work, group, call, mail, chat, location, globe, translate, notes: Material Symbols, accent-tinted).
- Added the semi-automatic duplicate remover, its own app-bar icon next to the overflow menu: the bridge scans every card for exact normalized matches (emails lowercased, phones compared on their trailing digits so country codes don't split a match, full names casefolded), unions them into groups, and skips cards that are already one contact. The review is one-shot: each tap surfaces the first unhandled group as a dialog of tappable card rows (name over addressbook; tapping one leaves the review and opens that card), with three verbs: Ignore remembers the dismissal in the store and never proposes the group again, Merge runs the normal merge flow (survivor choice, union form or direct absorb), and Link, offered only when every card lives in its own addressbook, makes the cards one contact by staging a shared UID on them. Tap again for the next group, until "No duplicates found".

- Wired the remaining form-worthy vCard 4.0 properties into the editor: Anniversary and Gender (sex code spinner plus free-text identity) under Identity, a Relations section (typed like phones: spouse, child, parent, sibling, friend, colleague, emergency), a Messaging section (IMPP URIs) and a Languages section (RFC 5646 tags). All ride the field model end to end (projection, patch, union merge, dedup, conflict view) and now count toward the divergence hash, so the store schema bumped to force a re-index.
- Added an advanced raw-property editor for power users, behind a new app-bar icon on the edit form: every line of the vCard as a tappable row (edit the raw text, remove it, or append a new property; each change reparses through the bridge and rejects invalid lines) with the full source at the bottom. It opens on single-card contacts only (raw lines cannot fan out to several documents), the form state is baked into the document on the way in and the form rebuilds from it on the way back, and SAVE stages the edited document even when the managed content is unchanged.

### Fixed

- Conflicts whose three-way merge needs no user choice now resolve during the sync itself instead of sitting flagged until tapped: the sync captures the remote side and runs the merge right away, stages the clean resolutions (counted as merged) and pushes them in a second reconcile within the same pass, so only genuine same-field collisions surface as conflicts and the diverged flag always opens a resolution form with something to ask. Before this, tapping such a card resolved it silently and just cleared the flag, which read as a bug; that tap-time path remains as a fallback and now says what it did with a toast.
- Fixed the JMAP backend resurrecting a display name on cards that carry none: editing such a contact server-side brought it back with an FN. The ContactCard conversion ran on calcard, whose JSContact-to-vCard export follows the RFC 9555 letter and derives a DERIVED=true FN from the name components whenever name.full is absent (vCard formally requires one), colliding with the app's no-display-name-minting rule. The jmap module now converts through vcard-rs's own RFC 9555 codec (the jscontact feature), which maps FN strictly to and from name.full and preserves everything unmapped through the vCardProps/JSPROP escape hatches; calcard left the bridge entirely. Cards already carrying a minted FN;DERIVED=TRUE heal on their next remote change or edit.
- The display name became hub-only on the phone spoke: the projection writes the structured name parts alone, each in its own column, the provider composing the phone-visible name from them, and never DISPLAY_NAME; the read-back ingests a DISPLAY_NAME only from a parts-free row (the shape apps write when only a full name is typed). ContactsProvider owns DISPLAY_NAME too tightly for anything else: it derives the visible name from the parts whenever present and rewrites DISPLAY_NAME as their join, so a projected FN either was re-split into the wrong parts by guesswork (FN projected alone, losing the real first and last on the phone) or vanished under the join and read back as a phone edit clearing the FN in the hub (FN projected alongside the parts). A Cardamum-side display name now simply never reaches the phone. Tapping a conflicted contact whose divergence needs no user choice also says so with a toast now, instead of silently clearing the flag and staying on the list.
- Fixed the zero-registration OAuth grant failing its authorization on fastmail. The root cause was a missing RFC 8707 resource indicator: fastmail's authorization endpoint bounces a request without a resource parameter back as invalid_target (masked as InvalidRequest until io-oauth learned the code), so the issuer flow now sends the config's plain endpoint URL (the JMAP session URL or CardDAV root the token is for) as the resource. The flow also stopped sending Google's provider-specific hints (access_type, prompt), the refresh token coming from the offline_access scope instead, while the standard login_hint prefill stays (fastmail tolerates it, verified live). The dynamic-registration redirect also took its RFC 8252 §7.1 shape, a single slash after the private-use scheme (org.pimalaya.cardamum:/oauth2redirect), and the random CSRF state and PKCE verifier are now drawn from letters and digits only, a strict subset of the RFC 7636 unreserved set. The issuer flow logs its registration, authorization URL and (code-redacted) redirect under the cardamum logcat tag, and unit tests pin the authorization URL wire shape and the single-slash redirect parsing.
- The app no longer mints a display name into cards: FN persists only when the card actually carries one (emptying the field drops it), and the lists compose one on the fly instead (name parts, else the first nickname, email or phone). The editor's display name also left the combined name row for its own Identity line, the Name row keeping only the five structured parts.
- A store schema rebuild no longer strands the app: upgrades now keep the addressbook table (syncs walk the stored addressbooks, so dropping them left connected accounts with nothing to sync from and empty screens), and the remote sync self-heals by re-fetching the addressbooks of any account missing them, all-subscribed.
- A conflict on a field only one card carries no longer opens an empty resolve view: the empty state now rides along as a pickable "Not set" chip next to the value, for every single-valued field.

### Changed

- Moved the offline driver's sync policy and the store's placement codec out of Java and into the bridge's new store module, leaving the Java side to gather row facts and execute SQL. The placement codec (row to engine placement on both axes, engine upsert back to row writes, the phone drop split) rides four plan verbs; the push planning (membership patch vs create or delete, the Google myContacts post-create patch), the account-wide delta projection onto one book and the CardDAV If-Match quirk retry heuristic are bridge decisions too; syncCards now handles the expired-cursor re-run and the initial CardDAV enumeration fallback internally, reporting completeness on the delta; and an unknown merge base (never captured) is the bridge's rule inside mergeCardChanges. The raw control bytes hiding in Java string literals (a NUL in the storage key, unit separators in the advanced editor) became unicode escapes along the way.
- Moved the edit form's vCard semantics and the merged-view identity rules out of Java and into the bridge's projection module. The form's type tables (phone, email/address, relation TYPE sets and GENDER sex codes) and their matching logic, the name/organization/address/gender summaries, the picker date round-trip and the typed-entry construction now come from two view-support verbs (formView computing the whole page's summaries, spinner positions and parsed dates in one pass; formEntry resolving a spinner position back into a typed entry) plus formDate; the Java form only builds views and localizes the labels the positions address. ### Fixed

- Fixed a push conflict deadlocking sync forever: when the remote changed under a staged local change, the 412 left the change staged, the staged row blocked every fetch from landing the remote edit, and each sync reported "1 pulled, 1 merged" without ever converging (a remote rename was invisible in the app indefinitely). The conflict now resolves with the vcard-rs three-way merge against the staged base: the local side wins same-field collisions, every other remote change flows in, and the merged card pushes against the fresh server state (unguarded on a second 412, the no-internal-ETag quirk). A staged delete conflicting with a remote update is dropped and the updated card adopted (an update beats a removal, nothing is silently lost). The "merged" sync counter now means conflicts actually reconciled.

- Fixed merging cards that already carry the same content (typically one merged contact's copies) opening the merge form for nothing: the absorb now happens directly after the survivor choice, form-free.
- Fixed the merge form keeping duplicated list entries: the bridge union now collapses model list entries that duplicate an earlier one (matched on their identifying fields, case-insensitively: the same number under two types prefills as one phone). Merging reformats, unlike faithful single-card editing.
- Fixed the selection checkboxes sitting off the app-bar icon column: they now ride centred in a 48dp trailing slot with the same 4dp end padding as the bar.
- Fixed rows growing slightly taller in selection mode (and on diverged rows): the trailing slot was 48dp tall, exceeding the avatar; it now hugs its content, so showing the checkbox or the conflict flag never changes the row height. The conflict flag also mirrors the checkbox's exact 32dp box and centre.
- Fixed the merge form's conflict chips hiding the prefilled value: every distinct choice now shows, the current one included, so all the conflicting values stay visible side by side.
- Fixed unsubscribing an addressbook dropping its cached cards, so toggling it off and on left the list empty until a remote sync. Subscription is now a pure view filter: the cards stay in the store, hidden and unsynced while unchecked, and reappear instantly on re-check.
- Fixed the merged-row placement line counting replicas instead of addressbooks: two linked cards of the same addressbook read "In 2 addressbooks". The line now counts distinct addressbooks and divergence became its own count-free "Diverged" flag (it can happen at any book count).
- Fixed connecting a new account deselecting every other account's addressbooks: the post-connect selection replaced the global subscription set, unsubscribing everything it did not mention. Subscriptions are now set per account.
- Fixed the selection-mode app-bar icons sitting one icon-width away from the edge: hiding the sync button left its 48dp spinner slot visible, pushing the link and delete icons left. The whole slot now hides with it.
- Fixed the HTTP 500 Microsoft Graph returned when pushing a modified contact. The update PATCHed a full-state body whose explicit nulls (fields the card never carried) the Outlook backend rejects with an internal server error, the same way it already rejected them on create. The push now diffs the edited vCard against the card's last-synced state and sends only the fields the edit changed, nulling just the ones it actually cleared; the store captures that base vCard on the first edit of a clean card and keeps it until a push confirms.

The bridge's default resolver is now `https://cloudflare-dns.com/dns-query` and the app no longer feeds the device's DNS server to the search: DoH rides the same TLS transport as every other request, so the SRV, TXT and MX lookups (and with them PACC and the JMAP SRV discovery) work on mobile networks that block outbound DNS over TCP, instead of being silently skipped. Everything stays coroutine-driven through the app's own transport; no pimconf client or TLS feature is pulled in.
- Drove the configuration proposals from the pimconf unified search instead of the CardDAV-only discovery: the connection flow now collects CardDAV and JMAP service configs (fixed provider rules, PACC, RFC 6764 CardDAV resolve, RFC 8620 JMAP resolve) and lists one radio line per protocol and authentication variant, so only protocols the server actually speaks are proposed (a domain without JMAP no longer shows a JMAP entry). Password lines carry the endpoint host and connect against the config's endpoint (JMAP behind its sentinel) with the config's username when one is advertised; OAuth variants stay disabled until shared client ids exist; a search finding nothing lands on the config screen with an explanatory disabled line instead of a blocking error.
- Reworked the app around multiple accounts and a home screen. `SecureStore` now holds a list of accounts (one encrypted blob keyed by email) and `CardStore` (schema v4) tags every addressbook with its account and a subscription flag. The landing screen lists the subscribed addressbooks grouped by account; tapping one opens its contacts (the app-bar dropdown is gone, replaced by a back arrow plus the addressbook name, and the settings cog is dropped). A filter icon opens a subscription editor (every addressbook, subscribed or not, with a checkbox and a Cancel/Save bar); connecting an account subscribes all of its addressbooks by default. The home's add button reopens the connection flow to add a further account (with a back arrow, unlike the first-run onboarding). The phone-contacts projection ("Sync local") stays independent from subscription and reconciles the Android accounts across every account at once. The Sync menu is now a two-line popup (cloud-upload "Sync remote", database "Sync local", each with a one-line description).
- Added the Google CardDAV OAuth 2.0 sign-in: the config screen's "Google CardDAV (OAuth)" entry now runs an authorization-code grant with PKCE (via the client's `OauthSession`), opening the authorization URL in the browser and catching the redirect on the reversed-client-id custom scheme (a `singleTop` intent-filter routed through `onNewIntent`). The access token stands in for the password (empty login, Bearer auth) and is stored the same encrypted way as a password; the display email is kept alongside so the per-addressbook Android accounts are still named after the user even though the login is empty. Google's own CardDAV endpoint is reused by the existing io-webdav path, so no People/Graph backend is needed. Client/redirect/endpoint/scope defaults live in an `Oauth` holder for now (bound for pimconf so every Pimalaya app shares them).
- Turned the contacts app-bar title into the addressbook name over the account email (the email at 0.8rem), added a search action between sync and settings that reveals a full-width filter bar matching raw vCard content with a close affordance, and standardized the app-bar padding so the selection checkboxes line up with the app-bar icons. List rows show the name at 1.1rem over email/phone at 0.8rem.

- Scaffolded the project: a Rust JNI bridge (`libcardamum.so`), the Android `:client` library owning the native boundary, and the `:app` module, all in plain Java and XML framework Views.
- Added the connection flow shown on first launch: the email's provider family is detected (Google, Microsoft, other), matching configurations are proposed, standard CardDAV is resolved via pimconf RFC 6764 discovery (SRV, TXT, .well-known) in the Rust bridge, the password is entered, the connection is verified, and the account is saved encrypted in the Android Keystore.

  Discovery prefers the active network's own DNS server for the SRV/TXT lookups and retries with the `.well-known` probe alone when the resolver is unreachable, since mobile networks routinely block outbound DNS-over-TCP to third parties. Failed transport upcalls surface the underlying Java exception class and message instead of a generic JNI error.
- Added the addressbook selection step after the connection check: the discovered addressbooks are listed with checkboxes and each selected one becomes its own Android account of Cardamum's account type (stub authenticator plus contacts sync-adapter registration), the collection URL living in the account's user data; deselecting an addressbook removes its account and ContactsProvider purges its contacts.
- Worked around servers whose If-Match always fails on some resources (posteo's SabreDAV serves an ETag in listings but some older resources carry none internally, so their preconditioned writes 412 forever): when the push is rejected but the listing fetched moments earlier still matches the staged base, the remote is provably unchanged and the write retries without If-Match; a genuine mismatch stays a counted conflict, now logged with both ETags.
- Keyed the phone projection on a hash of the vCard (raw contact SYNC4, salted with a projection version so mapping changes rewrite every contact once) instead of the ETag, so staged local edits project into the phone's contacts without waiting for a push; the ETag stays informational in SYNC1, since it is the remote spoke's revision marker, not the phone's.
- Made the vCard's FN always win as the phone-visible name: ContactsProvider derives the display name from the structured name components whenever they are present, so when FN diverges from the natural join of N's parts (e.g. only the display name was edited), the projection writes FN alone instead of letting the stale components override it.

- Split the two sync spokes behind a Sync menu on the contacts screen: "Sync remote" reconciles the store with the server (push staged changes, fetch; no phone access needed), "Sync local" reconciles the accounts and projects the store into the phone's contacts (local only, asks for the contacts permission on first use). - Added the sync screen, currently full-only and one-way: it fetches every selected addressbook and its cards from the remote, snapshots them into a local SQLite base the screens read from (falling back to that base when the remote is unreachable), and projects each addressbook into its Android account's raw contacts. Projection maps the vCard through a neutral field model built by vcard-rs in the Rust bridge (well-defined properties only, no custom X-* in either direction), writes with CALLER_IS_SYNCADAPTER so nothing is marked dirty, keeps the card id and ETag in the raw contact's sync columns, skips unchanged ETags, and never touches contacts of other accounts. - Made editing offline first: saving and deleting only stage the change in the SQLite base (dirty and tombstone flags), and the sync pushes staged changes (create If-None-Match, update and delete If-Match) against the card's server-returned resource name; a 412 leaves the change staged and surfaces a conflict count, and the remote is never contacted outside a sync.

  Each sync fetches before pushing, so staged rows learn the server's resource name first: cards created by other clients need not be named `<id>.vcf`, and reconstructing that path made every update of such a card fail with a spurious 412 (fixed together with io-webdav, which now carries the raw resource name in `CardEntry`/`CardRef`).

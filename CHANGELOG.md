## [Unreleased]

### Added

- Wired the remaining form-worthy vCard 4.0 properties into the editor: Anniversary and Gender (sex code spinner plus free-text identity) under Identity, a Relations section (typed like phones: spouse, child, parent, sibling, friend, colleague, emergency), a Messaging section (IMPP URIs) and a Languages section (RFC 5646 tags). All ride the field model end to end (projection, patch, union merge, dedup, conflict view) and now count toward the divergence hash, so the store schema bumped to force a re-index.
- Added an advanced raw-property editor for power users, behind a new app-bar icon on the edit form: every line of the vCard as a tappable row (edit the raw text, remove it, or append a new property; each change reparses through the bridge and rejects invalid lines) with the full source at the bottom. It opens on single-card contacts only (raw lines cannot fan out to several documents), the form state is baked into the document on the way in and the form rebuilds from it on the way back, and SAVE stages the edited document even when the managed content is unchanged.

### Fixed

- A store schema rebuild no longer strands the app: upgrades now keep the addressbook table (syncs walk the stored addressbooks, so dropping them left connected accounts with nothing to sync from and empty screens), and the remote sync self-heals by re-fetching the addressbooks of any account missing them, all-subscribed.
- A conflict on a field only one card carries no longer opens an empty resolve view: the empty state now rides along as a pickable "Not set" chip next to the value, for every single-valued field.

### Fixed

- Fixed a push conflict deadlocking sync forever: when the remote changed under a staged local change, the 412 left the change staged, the staged row blocked every fetch from landing the remote edit, and each sync reported "1 pulled, 1 merged" without ever converging (a remote rename was invisible in the app indefinitely). The conflict now resolves with the vcard-rs three-way merge against the staged base: the local side wins same-field collisions, every other remote change flows in, and the merged card pushes against the fresh server state (unguarded on a second 412, the no-internal-ETag quirk). A staged delete conflicting with a remote update is dropped and the updated card adopted (an update beats a removal, nothing is silently lost). The "merged" sync counter now means conflicts actually reconciled.

- Fixed merging cards that already carry the same content (typically one merged contact's copies) opening the merge form for nothing: the absorb now happens directly after the survivor choice, form-free.
- Fixed the merge form keeping duplicated list entries: the bridge union now collapses model list entries that duplicate an earlier one (matched on their identifying fields, case-insensitively: the same number under two types prefills as one phone). Merging reformats, unlike faithful single-card editing.
- Fixed the selection checkboxes sitting off the app-bar icon column: they now ride centred in a 48dp trailing slot with the same 4dp end padding as the bar.
- Fixed rows growing slightly taller in selection mode (and on diverged rows): the trailing slot was 48dp tall, exceeding the avatar; it now hugs its content, so showing the checkbox or the conflict flag never changes the row height. The conflict flag also mirrors the checkbox's exact 32dp box and centre.
- Fixed the merge form's conflict chips hiding the prefilled value: every distinct choice now shows, the current one included, so all the conflicting values stay visible side by side.
- Fixed unsubscribing an addressbook dropping its cached cards, so toggling it off and on left the list empty until a remote sync. Subscription is now a pure view filter: the cards stay in the store, hidden and unsynced while unchecked, and reappear instantly on re-check.
- Fixed the merged-row placement line counting replicas instead of addressbooks: two linked cards of the same addressbook read "In 2 addressbooks". The line now counts distinct addressbooks and divergence became its own count-free "Diverged" flag (it can happen at any book count).
- Fixed connecting a new account deselecting every other account's addressbooks: the post-connect selection replaced the global subscription set, unsubscribing everything it did not mention. Subscriptions are now set per account.
- Fixed the selection-mode app-bar icons sitting one icon-width away from the edge: hiding the sync button left its 48dp spinner slot visible, pushing the link and delete icons left. The whole slot now hides with it.
- Fixed the HTTP 500 Microsoft Graph returned when pushing a modified contact. The update PATCHed a full-state body whose explicit nulls (fields the card never carried) the Outlook backend rejects with an internal server error, the same way it already rejected them on create. The push now diffs the edited vCard against the card's last-synced state and sends only the fields the edit changed, nulling just the ones it actually cleared; the store captures that base vCard on the first edit of a clean card and keeps it until a push confirms.

The contacts list now renders entirely from those columns and the app-side vCard mini-parser is gone (the app no longer parses a single vCard in Java; one-off lookups like the editor title go through the same bridge call). Checking a book adds the contact there: a staged membership change on the replica's own JMAP or Google account (pushed as one addressBookIds patch, or one contactGroups members:modify per group), a copied replica with the same UID (instantly linked in the merged view) on any other account or CardDAV/Graph collection. Unchecking removes it: a staged membership removal when the replica keeps other books, a staged delete of the replica otherwise; the addressbook the editor was opened from is locked. Membership changes stage offline like everything else (schema v7 adds a state to memberships; opposite stagings cancel out) and push after the cards, so a copied replica's group memberships follow its server-assigned id, fixing the stage 2 limitation where a Google copy could only land in myContacts.

- Made the Google People and Microsoft Graph backends preserve the vCard data they cannot express natively (docs/custom-data.md). Every property line that does not project onto a native field (X-* properties, standard properties without a provider slot, values past a Graph slot, partial birthdays) is stashed verbatim server-side (a Google clientData entry keyed cardamum.vcard; a Graph single-value extended MAPI property riding inline in the same requests, read back via a filtered $expand) and restored byte for byte on fetch, so the document of record survives Google and Microsoft round trips with no loss; lines over 8 KiB (photo blobs) stay local-only. Google stash writes merge foreign clientData entries under the person-etag guard so other clients' data is never clobbered. In the other direction, provider-only fields now ride the vCard as read-only vendor properties, re-minted from server truth on every fetch and consumed on push: Google group memberships, external ids, misc keywords and locations as X-GOOGLE-*; Graph fileAs, officeLocation, assistantName and manager as X-MSGRAPH-*. A card moved to another addressbook carries all of it as plain vCard lines.
- Added the Google People API backend behind the same client operations as CardDAV, Microsoft Graph and JMAP. The config screen's disabled "Google Contacts API" placeholder became two working options (the shared Cardamum OAuth client and a custom one), running the same PKCE authorization-code grant as the Google CardDAV entry but with the People contacts scope; the connected account carries the `google://<email>` sentinel base URL, which routes every operation through io-google-people's coroutines in the Rust bridge, with a new google module projecting People person resources to and from the vCard document of record (typed emails/phones/addresses, org and title, birthday, biography as NOTE, spouse and child relations as RELATED). - Added the JMAP for Contacts (RFC 9610) backend behind the same client operations as CardDAV and Microsoft Graph. Accounts whose base URL carries the `jmap://<host>[/path]` sentinel talk to the HTTPS session endpoint behind it (a bare host goes through `/.well-known/jmap` discovery); the Rust bridge runs io-jmap's RFC 9610 coroutines over the existing Java transport and converts ContactCards (JSContact, RFC 9553) to and from the vCard document of record via calcard (RFC 9555). AddressBooks list with their id appended to the base URL, cards address by their ContactCard id, the ETag surfaced to the store is a hash of the card's JSON (JMAP has no per-card ETag), and updates PATCH only the properties that differ from the last-synced base vCard (last-write-wins, like Graph).
- Added an API token option to the configuration screen for services whose endpoint advertises Bearer authentication (detected by the pimconf JMAP session probe): the token is sent as `Authorization: Bearer` (the empty-login convention) instead of Basic credentials. Fastmail's JMAP API only accepts bearer API tokens, so a valid token entered as a password used to 401.
- Fixed the 404 when connecting to a CardDAV endpoint advertised as a bare origin (e.g. PACC's carddav.fastmail.com, which rejects every request outside /dav/*): the addressbook walk now probes /.well-known/carddav first (RFC 6764 §5) and starts from its redirect target, falling back to the origin when the probe finds nothing.
- Switched the discovery DNS lookups to RFC 8484 DNS-over-HTTPS. The bridge's default resolver is now `https://cloudflare-dns.com/dns-query` and the app no longer feeds the device's DNS server to the search: DoH rides the same TLS transport as every other request, so the SRV, TXT and MX lookups (and with them PACC and the JMAP SRV discovery) work on mobile networks that block outbound DNS over TCP, instead of being silently skipped. Everything stays coroutine-driven through the app's own transport; no pimconf client or TLS feature is pulled in.
- Drove the configuration proposals from the pimconf unified search instead of the CardDAV-only discovery: the connection flow now collects CardDAV and JMAP service configs (fixed provider rules, PACC, RFC 6764 CardDAV resolve, RFC 8620 JMAP resolve) and lists one radio line per protocol and authentication variant, so only protocols the server actually speaks are proposed (a domain without JMAP no longer shows a JMAP entry). Password lines carry the endpoint host and connect against the config's endpoint (JMAP behind its sentinel) with the config's username when one is advertised; OAuth variants stay disabled until shared client ids exist; a search finding nothing lands on the config screen with an explanatory disabled line instead of a blocking error.

- Scaffolded the project: a Rust JNI bridge (`libcardamum.so`), the Android `:client` library owning the native boundary, and the `:app` module, all in plain Java and XML framework Views.
- Added the connection flow shown on first launch: the email's provider family is detected (Google, Microsoft, other), matching configurations are proposed, standard CardDAV is resolved via pimconf RFC 6764 discovery (SRV, TXT, .well-known) in the Rust bridge, the password is entered, the connection is verified, and the account is saved encrypted in the Android Keystore.

  Discovery prefers the active network's own DNS server for the SRV/TXT lookups and retries with the `.well-known` probe alone when the resolver is unreachable, since mobile networks routinely block outbound DNS-over-TCP to third parties. Failed transport upcalls surface the underlying Java exception class and message instead of a generic JNI error.

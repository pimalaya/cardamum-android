# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Fixed

- Fixed the HTTP 500 Microsoft Graph returned when pushing a modified contact. The update PATCHed a full-state body whose explicit nulls (fields the card never carried) the Outlook backend rejects with an internal server error, the same way it already rejected them on create. The push now diffs the edited vCard against the card's last-synced state and sends only the fields the edit changed, nulling just the ones it actually cleared; the store captures that base vCard on the first edit of a clean card and keeps it until a push confirms.

### Added

- Added the JMAP for Contacts (RFC 9610) backend behind the same client operations as CardDAV and Microsoft Graph. Accounts whose base URL carries the `jmap://<host>[/path]` sentinel talk to the HTTPS session endpoint behind it (a bare host goes through `/.well-known/jmap` discovery); the Rust bridge runs io-jmap's RFC 9610 coroutines over the existing Java transport and converts ContactCards (JSContact, RFC 9553) to and from the vCard document of record via calcard (RFC 9555). AddressBooks list with their id appended to the base URL, cards address by their ContactCard id, the ETag surfaced to the store is a hash of the card's JSON (JMAP has no per-card ETag), and updates PATCH only the properties that differ from the last-synced base vCard (last-write-wins, like Graph).
- Added an API token option to the configuration screen for services whose endpoint advertises Bearer authentication (detected by the pimconf JMAP session probe): the token is sent as `Authorization: Bearer` (the empty-login convention) instead of Basic credentials. Fastmail's JMAP API only accepts bearer API tokens, so a valid token entered as a password used to 401.
- Fixed the 404 when connecting to a CardDAV endpoint advertised as a bare origin (e.g. PACC's carddav.fastmail.com, which rejects every request outside /dav/*): the addressbook walk now probes /.well-known/carddav first (RFC 6764 §5) and starts from its redirect target, falling back to the origin when the probe finds nothing.
- Switched the discovery DNS lookups to RFC 8484 DNS-over-HTTPS. The bridge's default resolver is now `https://cloudflare-dns.com/dns-query` and the app no longer feeds the device's DNS server to the search: DoH rides the same TLS transport as every other request, so the SRV, TXT and MX lookups (and with them PACC and the JMAP SRV discovery) work on mobile networks that block outbound DNS over TCP, instead of being silently skipped. Everything stays coroutine-driven through the app's own transport; no pimconf client or TLS feature is pulled in.
- Drove the configuration proposals from the pimconf unified search instead of the CardDAV-only discovery: the connection flow now collects CardDAV and JMAP service configs (fixed provider rules, PACC, RFC 6764 CardDAV resolve, RFC 8620 JMAP resolve) and lists one radio line per protocol and authentication variant, so only protocols the server actually speaks are proposed (a domain without JMAP no longer shows a JMAP entry). Password lines carry the endpoint host and connect against the config's endpoint (JMAP behind its sentinel) with the config's username when one is advertised; OAuth variants stay disabled until shared client ids exist; a search finding nothing lands on the config screen with an explanatory disabled line instead of a blocking error.
- Reworked the app around multiple accounts and a home screen. `SecureStore` now holds a list of accounts (one encrypted blob keyed by email) and `CardStore` (schema v4) tags every addressbook with its account and a subscription flag. The landing screen lists the subscribed addressbooks grouped by account; tapping one opens its contacts (the app-bar dropdown is gone, replaced by a back arrow plus the addressbook name, and the settings cog is dropped). A filter icon opens a subscription editor (every addressbook, subscribed or not, with a checkbox and a Cancel/Save bar); connecting an account subscribes all of its addressbooks by default. The home's add button reopens the connection flow to add a further account (with a back arrow, unlike the first-run onboarding). The phone-contacts projection ("Sync local") stays independent from subscription and reconciles the Android accounts across every account at once. The Sync menu is now a two-line popup (cloud-upload "Sync remote", database "Sync local", each with a one-line description).
- Added the Google CardDAV OAuth 2.0 sign-in: the config screen's "Google CardDAV (OAuth)" entry now runs an authorization-code grant with PKCE (via the client's `OauthSession`), opening the authorization URL in the browser and catching the redirect on the reversed-client-id custom scheme (a `singleTop` intent-filter routed through `onNewIntent`). The access token stands in for the password (empty login, Bearer auth) and is stored the same encrypted way as a password; the display email is kept alongside so the per-addressbook Android accounts are still named after the user even though the login is empty. Google's own CardDAV endpoint is reused by the existing io-webdav path, so no People/Graph backend is needed. Client/redirect/endpoint/scope defaults live in an `Oauth` holder for now (bound for pimconf so every Pimalaya app shares them).
- Turned the contacts app-bar title into the addressbook name over the account email (the email at 0.8rem), added a search action between sync and settings that reveals a full-width filter bar matching raw vCard content with a close affordance, and standardized the app-bar padding so the selection checkboxes line up with the app-bar icons. List rows show the name at 1.1rem over email/phone at 0.8rem.

- Scaffolded the project: a Rust JNI bridge (`libcardamum.so`), the Android `:client` library owning the native boundary, and the `:app` module, all in plain Java and XML framework Views.
- Added the connection flow shown on first launch: the email's provider family is detected (Google, Microsoft, other), matching configurations are proposed, standard CardDAV is resolved via pimconf RFC 6764 discovery (SRV, TXT, .well-known) in the Rust bridge, the password is entered, the connection is verified, and the account is saved encrypted in the Android Keystore.

  Discovery prefers the active network's own DNS server for the SRV/TXT lookups and retries with the `.well-known` probe alone when the resolver is unreachable, since mobile networks routinely block outbound DNS-over-TCP to third parties. Failed transport upcalls surface the underlying Java exception class and message instead of a generic JNI error.
- Added the addressbook selection step after the connection check: the discovered addressbooks are listed with checkboxes and each selected one becomes its own Android account of Cardamum's account type (stub authenticator plus contacts sync-adapter registration), the collection URL living in the account's user data; deselecting an addressbook removes its account and ContactsProvider purges its contacts.
- Worked around servers whose If-Match always fails on some resources (posteo's SabreDAV serves an ETag in listings but some older resources carry none internally, so their preconditioned writes 412 forever): when the push is rejected but the listing fetched moments earlier still matches the staged base, the remote is provably unchanged and the write retries without If-Match; a genuine mismatch stays a counted conflict, now logged with both ETags.
- Keyed the phone projection on a hash of the vCard (raw contact SYNC4, salted with a projection version so mapping changes rewrite every contact once) instead of the ETag, so staged local edits project into the phone's contacts without waiting for a push; the ETag stays informational in SYNC1, since it is the remote spoke's revision marker, not the phone's.
- Made the vCard's FN always win as the phone-visible name: ContactsProvider derives the display name from the structured name components whenever they are present, so when FN diverges from the natural join of N's parts (e.g. only the display name was edited), the projection writes FN alone instead of letting the stale components override it.
- Extracted the neutral-field-model to ContactsContract mapping into a pure `Mapping` class covered by JVM unit tests (`gradle :app:testDebugUnitTest`), pinning every mimetype, column and type constant so a mapping regression (like the FN-vs-components one) breaks in CI instead of silently on a phone.
- Rebuilt the contact edit form: sections (Name, Contact, Address, Work, More, Source) under horizontally scrollable tabs with a chevron that appears only while the strip can scroll further, a grey label above every field aligned to the field's text start with a brand-tinted underline (no placeholders, so a filled form stays readable), a small round add (+) icon by each multi-value label and a small round delete icon per row (borderless, glyph-aligned to their content), the native date picker for the birthday, a back arrow in the app bar, and a full-width fixed Save.
- Gave the contacts list ripple press feedback, hash-coloured letter avatars, no separators, and a sticky first-letter gutter; long-press enters a multi-select mode (checkboxes, "N selected" title, a delete icon in the app bar that removes the selection after confirmation, device-back to exit).
- Replaced the icon set with Material Design icons (Apache-2.0), traced as tintable vector drawables: arrow_back, add, remove, delete, sync (double arrow), person_add, chevron_right, close. No launcher or dependency change.
- Themed the whole UI from the device theme: the app bar takes the phone's colorPrimary, accents (field underlines, add icons, the sticky letter) take colorAccent, labels and icons take the standard text colours, and a DeviceDefault day/night base makes the app follow the system light/dark setting automatically. Only two neutral surface colours remain hard-coded, each with a night variant.
- Polished the contacts and form screens: the sync icon is the double-arrow sync glyph, the tab strip has a surface background and its scroll chevron is visible from the start (re-evaluated after layout) instead of appearing only mid-scroll, list rows have even padding and their checkboxes line up under the app-bar icons, selection mode gains a close (X) affordance to cancel, and explicit font sizes were dropped so every label uses the theme standard.
- Switched to a framework ListView so the contacts list recycles its rows (no lag with 150+ contacts), each row showing the name over a diminished first-email/first-phone line and a muted hash-coloured avatar; the add-contact action moved to a floating button and its app-bar slot became a settings cog opening a (blank for now) settings screen.
- Adopted Material Symbols (the Material Design 3 icon set from fonts.google.com/icons) in place of the classic Material icons, added a left tab chevron mirroring the right one, gave the edit form even spacing between field groups with one always-present empty row per multi-value field.
- Turned the contacts app-bar title into an addressbook dropdown that sizes to its content and carries the standard action-bar title appearance (the list and the add button follow the chosen addressbook), split each row's subtitle into a first-email then first-phone line, both smaller than the name, with the name in the standard bright text and the sub-lines diminished, enlarged the avatar letter, shrank the field labels to the small text appearance and moved the section plus button inline next to its label, and switched the row add/remove glyphs to a plain plus and close. The soft keyboard is now dismissed when leaving the edit form.
- Gave the floating add button the accent colour with its icon tinted to contrast the accent's luminance, so it stands out from the window and stays legible in both light and dark themes.
- Made the per-addressbook accounts appear under the accounts section of contacts apps: they are marked syncable for the contacts authority (isAlwaysSyncable plus an explicit setIsSyncable at reconcile time), and the sync service now declares the CONTACTS_STRUCTURE edit schema contacts apps require to recognize an account type as a contacts source (the DAVx5 mechanism; note the AOSP parser hard-requires the phonetic name attributes and a photo kind, and silently drops the account type otherwise, logging a DefinitionException). Cardamum creates no groups: the labels section of contacts apps belongs to the user, and any group left over from earlier builds is removed on the next local sync; vCard CATEGORIES stay vCard-side.
- Moved the store-to-phone projection into the sync adapter and pointed "Sync local" at Android's sync scheduler (one manual expedited requestSync per account), so the system's per-account "sync now" runs the same code; automatic sync starts off.
- Stopped resetting the user's automatic-sync choice: off is only the default at account creation, and the reconcile that precedes every sync no longer touches it (it used to force it back off, silently undoing the toggle in the system settings).
- Split the two sync spokes behind a Sync menu on the contacts screen: "Sync remote" reconciles the store with the server (push staged changes, fetch; no phone access needed), "Sync local" reconciles the accounts and projects the store into the phone's contacts (local only, asks for the contacts permission on first use). Launch is offline first: the contacts screen renders instantly from the store and never touches the network by itself.
- Added the sync screen, currently full-only and one-way: it fetches every selected addressbook and its cards from the remote, snapshots them into a local SQLite base the screens read from (falling back to that base when the remote is unreachable), and projects each addressbook into its Android account's raw contacts. Projection maps the vCard through a neutral field model built by vcard-rs in the Rust bridge (well-defined properties only, no custom X-* in either direction), writes with CALLER_IS_SYNCADAPTER so nothing is marked dirty, keeps the card id and ETag in the raw contact's sync columns, skips unchanged ETags, and never touches contacts of other accounts. The three-way engine with phone-edit detection is specified in docs/design.md and docs/contacts-mapping.md.
- Added the contacts screen listing every synced addressbook's vCards by formatted name, with a tabbed form editor (name, contact, address, other, read-only vCard source) over the neutral field model; the Rust bridge patches the collected model onto the stored vCard through the vcard-rs CST, so properties the form does not manage survive untouched.
- Made editing offline first: saving and deleting only stage the change in the SQLite base (dirty and tombstone flags), and the sync pushes staged changes (create If-None-Match, update and delete If-Match) against the card's server-returned resource name; a 412 leaves the change staged and surfaces a conflict count, and the remote is never contacted outside a sync.

  Each sync fetches before pushing, so staged rows learn the server's resource name first: cards created by other clients need not be named `<id>.vcf`, and reconstructing that path made every update of such a card fail with a spurious 412 (fixed together with io-webdav, which now carries the raw resource name in `CardEntry`/`CardRef`).
- Added the Nix flake providing the Rust toolchain (Android targets), cargo-ndk, the Android SDK/NDK, JDK 17 and Gradle.
- Added the release workflow assembling one APK per ABI plus a universal one, signed from an environment-provided keystore when present.
